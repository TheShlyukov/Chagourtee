# Chagourtee

Самохостируемый мессенджер: владелец запускает сервер на своём компьютере, создаёт инвайты и верифицирует участников по кодовому слову. В перспективе — федерация между серверами.

[English version](./README.md) | [Версия на русском](./README.ru.md)

## Стек

- **Сервер:** Node.js, Fastify, SQLite (better-sqlite3), WebSocket (ws)
- **Клиент:** React, TypeScript, Vite

## Особенности

- **Самохостинг**: Полный контроль над своими данными, размещение на собственном сервере
- **Верификация пользователей**: Возможность верификации участников по кодовому слову
- **Реал-тайм чат**: Общение в режиме реального времени с поддержкой типинга и присутствия
- **Управление пользователями**: Роли владельца, модератора и участника
- **Безопасность**: Авторизация через сессии, хеширование паролей
- **Администрирование**: Панель управления для создания комнат и инвайтов
- **Редактирование сообщений**: Возможность редактировать и удалять свои сообщения
- **Группировка сообщений**: Визуальное объединение сообщений одного пользователя
- **Множественный выбор сообщений**: Для массового удаления

## Быстрый старт

### 1. Установка

```bash
npm install
```

### 2. Запуск в режиме разработки

В двух терминалах:

```bash
# Сервер (порт 3000)
npm run dev:server

# Клиент (порт 5173, проксирует /api и /ws на сервер)
npm run dev:client
```

Или одной командой:

```bash
npm run dev
```

Откройте http://localhost:5173

**Доступ с телефона или другого устройства в той же Wi‑Fi:** откройте в браузере `http://ИМЯ_ИЛИ_IP_ВАШЕГО_КОМПЬЮТЕРА:5173` (например, `http://macbook-dana.local:5173`). Чтобы ссылки-инвайты вели на этот адрес (а не на localhost), создайте в корне проекта или в `client/` файл `.env` и добавьте строку:
```bash
VITE_APP_PUBLIC_URL=http://<ВАШ_IP_АДРЕСС>:5173
```
После изменения перезапустите клиент (`npm run dev:client`).

### 3. Первый владелец

Если в базе ещё нет пользователей, первого владельца можно создать так:

1. Установите переменную окружения (или добавьте в `server/config.example.env` и переименуйте в `.env`):
   ```bash
   export CHAGOURTEE_BOOTSTRAP_SECRET=ваш-секретный-ключ
   ```
2. На странице регистрации (/register) или вызовите API:
   ```bash
   curl -X POST http://localhost:3000/api/auth/register \
     -H "Content-Type: application/json" \
     -d '{"login":"admin","password":"ваш-пароль","bootstrap":"ваш-секретный-ключ"}'
   ```
3. После этого входите под созданным логином – вы владелец. Создайте инвайты в Админке и при необходимости комнаты.

### 4. Участники

- Владелец (или модератор) создаёт инвайт в Админке и передаёт ссылку вида:  
  `https://ваш-адрес/register?invite=КОД_ИНВАЙТА`
- Участник переходит по ссылке, вводит логин, пароль и при необходимости кодовое слово.
- Если указано кодовое слово, аккаунт остаётся в статусе «ожидает верификации» до тех пор, пока владелец в Админке не проверит слово и не нажмёт «Подтвердить».

## Система сборки

Проект использует npm workspaces с раздельными пакетами для клиента и сервера. Вот сводка доступных команд:

| Команда | Описание |
|---------|-------------|
| `npm run dev` | Запустить клиент и сервер в режиме разработки |
| `npm run dev:server` | Запустить только сервер (с горячей перезагрузкой через nodemon) |
| `npm run dev:client` | Запустить только клиент (Vite dev server на :5173) |
| `npm run build` | Собрать клиент и сервер для продакшена |
| `npm run build:client` | Собрать только клиент (в `client/dist/`) |
| `npm run build:server` | Упаковать сервер с esbuild (в `server/dist/`) |
| `npm start` | Запустить клиент (preview) и сервер из продакшен-сборок |
| `npm run start:server` | Запустить только собранный сервер |
| `npm run start:client` | Запустить только preview-сервер клиента |
| `npm run update` | Обновиться до последней версии из репозитория |
| `npm run db:backup` | Создать резервную копию SQLite базы в `data/backups/` |
| `npm run generate-encryption-key` | Сгенерировать ключ шифрования AES-256 для медиа |
| `npm run lint` | Запустить ESLint для клиентского кода (теперь настроен) |
| `npm run clear-logs` | Очистить файл логов |
| `npm run daemon:start` | Запустить сервер в фоне |
| `npm run daemon:stop` | Остановить запущенный сервер |
| `npm run daemon:restart` | Перезапустить сервер |
| `npm run daemon:status` | Проверить статус сервера |

### Детали упаковки сервера

Сервер упаковывается с помощью [esbuild](https://esbuild.github.io/) для более быстрого запуска и удобного деплоя:

- **Целевая платформа**: Компилируется под вашу текущую версию Node.js (v25.9.0) и платформу (darwin/arm64)
- **Нативные модули**: `better-sqlite3`, `bcrypt` и установщики FFmpeg содержат `.node` бинарные файлы, специфичные для платформы. Они сохраняются как внешние зависимости и автоматически копируются в `server/dist/node_modules/` при сборке
- **Директория данных**: Папка `data/` общая для режимов разработки и продакшена
- **Окружение**: Переменные загружаются из `server/.env` как обычно

**Важно:** Если вы деплоите на другую платформу (например, Linux-сервер), нужно пересобрать серверный бандл на этой платформе, чтобы получить корректные бинарники нативных модулей.

## Деплой у владельца

### Продакшен-сборка

```bash
# Собрать клиент и сервер
npm run build

# Или по отдельности:
npm run build:client   # собрать клиент в client/dist/
npm run build:server   # упаковать сервер с esbuild в server/dist/
```

Собранный клиент окажется в `client/dist/`. Упакованный сервер — в `server/dist/`.

**Примечание:** Сервер упаковывается с помощью [esbuild](https://esbuild.github.io/). Нативные модули (`better-sqlite3`, `bcrypt`, установщики FFmpeg) содержат платформенно-специфичные бинарники и сохраняются как внешние зависимости. Они автоматически копируются в `server/dist/node_modules/` при сборке.

### Запуск продакшен-сборок

```bash
# Запустить клиент (preview) и сервер
npm start

# Или по отдельности:
npm run start:server   # запустить собранный сервер из server/dist/
npm run start:client   # запустить preview-сервер клиента

# Примечание: Если продакшен-сборки не существует, npm start предложит собрать сначала
```

### Запуск сервера (без упаковки)

```bash
cd server
PORT=3000 node src/index.js
```

Рекомендуется задать переменные окружения (или файл `.env` в `server/`):

- `PORT` – порт (по умолчанию 3000)
- `HOST` – хост (0.0.0.0 для доступа с других машин)
- `CHAGOURTEE_DB_PATH` – путь к файлу SQLite (по умолчанию `./data/chagourtee.db`)
- `CHAGOURTEE_SESSION_SECRET` – секрет для cookie (обязательно смените)
- `CHAGOURTEE_BOOTSTRAP_SECRET` – секрет для создания первого владельца (см. выше)
- `CHAGOURTEE_MAX_FILE_SIZE` – максимальный размер файла для загрузки в байтах (по умолчанию 52428800 = 50MB)
- `CHAGOURTEE_MEDIA_ENCRYPTION_KEY` – ключ шифрования для медиафайлов (32 байта для AES-256)

### Логирование

Chagourtee поддерживает расширенную конфигурацию логирования:

- `CHAGOURTEE_LOG_TO_FILE` – Включить запись логов в файл (`true` или `false`, по умолчанию `false`)
- `CHAGOURTEE_LOG_LEVEL` – Уровень логирования: `error`, `warn`, `info`, `debug`, `trace` (по умолчанию `info`)
- `CHAGOURTEE_LOG_DIR` – Пользовательская директория для логов (опционально, по умолчанию `logs/` в корне проекта)

При `CHAGOURTEE_LOG_TO_FILE=true`, все логи записываются в `logs/chagourtee.log` в дополнение к выводу в консоль.

Для очистки логов:
```bash
npm run clear-logs
```

### Запуск в фоне

Используйте менеджер демонов для запуска собранных продакшен-билдов в фоне:

```bash
# Сначала соберите
npm run build

# Запустить сервер + клиент в фоне
npm run daemon:start

# Проверить статус
npm run daemon:status

# Остановить оба процесса
npm run daemon:stop

# Перезапустить оба
npm run daemon:restart

# Просмотр логов в реальном времени
tail -f logs/chagourtee.log
```

Менеджер демонов хранит PID-ы в `server/data/chagourtee-server.pid` и `server/data/chagourtee-client.pid`.

> **Примечание:** Демон работает только с продакшен-билдами. Для разработки используйте `npm run dev` в терминале.

### Доступ из интернета

Сервер за NAT/домашним роутером недоступен по внешнему IP без проброса портов. Варианты:

- **Проброс портов** на роутере на машину с сервером.
- **Tailscale / ZeroTier** – участники подключаются по выданному адресу в VPN.
- **Cloudflare Tunnel** – туннель с вашей машины в интернет по домену.

Клиент должен открываться по тому же origin, что и API (или настроен CORS и cookie с правильным domain).

### Резервное копирование БД

```bash
npm run db:backup
```

Копия SQLite сохраняется в `data/backups/` (или рядом с `CHAGOURTEE_DB_PATH` в подкаталоге `backups`).

## Обновление сервиса

Чтобы обновить вашу установку Chagourtee до последней доступной версии:

### Из корня проекта:
```bash
npm run update
```

### Или запустите напрямую из серверной части:
```bash
cd server && npm run update-service
```

Скрипт обновления выполнит следующие действия:
1. Проверит текущую версию вашей установки
2. Загрузит последние доступные теги из удаленного репозитория
3. Сравнит текущую версию с последней доступной версией
4. Предложит обновление, если доступна более новая версия
5. Выполнит обновление путем:
   - Загрузки последних изменений
   - Переключения на целевой тег версии
   - Установки зависимостей для всех рабочих пространств

> **Примечание**: Обязательно сохраните все локальные изменения перед запуском обновления, поскольку скрипт переключится на определенный тег, который может перезаписать несохраненные изменения.

## API (кратко)

- `POST /api/auth/login` – вход
- `POST /api/auth/logout` – выход
- `GET /api/auth/me` – текущий пользователь
- `POST /api/auth/register` – регистрация (inviteId или bootstrap)
- `GET/POST/PATCH/DELETE /api/rooms` – комнаты (CRUD для владельца/модератора)
- `GET /api/rooms/:id/messages`, `POST /api/rooms/:id/messages` – сообщения
- `GET /api/invites` – инвайты
- `GET /api/verification/pending`, `POST /api/verification/approve`, `reject` – верификация по кодовому слову
- `POST /api/profile/change-password`, `change-login`, `codeword` – профиль
- WebSocket `/ws` – новые сообщения, типинг, присутствие (после входа по cookie)

## Архитектура

### Клиентская сторона

Фронтенд реализован на React с использованием TypeScript и Vite. Приложение организовано по следующим компонентам:

- **AuthContext**: Управление аутентификацией и состоянием пользователя
- **Layout**: Основной макет приложения
- **Pages**: Компоненты страниц (Login, Register, Chat, Profile, Admin, VerificationWaiting)
- **API**: Модуль для взаимодействия с сервером
- **WebSocket**: Реализация реал-тайм коммуникации

### Серверная сторона

Backend построен на Fastify с использованием SQLite для хранения данных:

- **DB**: Подключение к базе данных через better-sqlite3
- **Auth**: Аутентификация и управление сессиями
- **Routes**: REST API для работы с комнатами, сообщениями, инвайтами и т.д.
- **WS**: WebSocket сервер для реал-тайм общения

### Безопасность

- Хеширование паролей с помощью bcrypt
- Защита сессий через cookies с настройками httpOnly и secure
- Проверка прав доступа к защищенным маршрутам
- Проверка верификации пользователя для определенных действий
- Защита от несанкционированного доступа к комнатам

## Поддержка медиафайлов

В последних обновлениях Chagourtee получил поддержку обмена медиафайлами в чате. Пользователи могут делиться:

- Изображениями (JPG, PNG, GIF, WebP, SVG, BMP, TIFF)
- Видео (MP4, WebM, OGG, MPEG, QuickTime)
- Аудиофайлами (MP3, WAV, AAC, OGG, MIDI, WebM, AIF, M4A, AIFF, ALAC, FLAC и др.)
- Документами (PDF, DOC, DOCX, XLS, XLSX, TXT, ZIP, RAR и др.)
- Любые другие типы файлов (все типы файлов поддерживаются)

### Транскодирование аудио

Chagourtee включает автоматическое транскодирование аудио для лучшей совместимости с браузерами. Когда загружаются определенные аудиоформаты, которые плохо поддерживаются во всех браузерах, система автоматически преобразует их в формат FLAC для воспроизведения:

- AIFF/AIFF файлы
- M4A файлы, закодированные с помощью ALAC (Apple Lossless)
- Другие аудиоформаты, которые могут вызвать проблемы с воспроизведением в некоторых браузерах

Когда эти файлы запрашиваются, система автоматически конвертирует их в формат FLAC для более широкой совместимости с браузерами. Транскодированная версия кэшируется и передается клиентам вместо исходного файла. Исходный файл остается без изменений и защищен.

**Примечание**: Для работы этой функции в системе должен быть установлен FFmpeg. На большинстве систем вы можете установить его с помощью:

```bash
# Ubuntu/Debian
sudo apt update && sudo apt install ffmpeg

# macOS с Homebrew
brew install ffmpeg

# Или установите ffmpeg-static как резервный вариант:
cd server && npm install ffmpeg-static
```

### Настройка

Для включения шифрования медиафайлов необходимо сгенерировать и настроить ключ шифрования:

1. Сгенерируйте новый ключ шифрования:
   ```bash
   cd server
   npm run generate-encryption-key
   ```

2. Скопируйте сгенерированный ключ в ваш файл `.env` как `CHAGOURTEE_MEDIA_ENCRYPTION_KEY`

3. Все загружаемые медиафайлы шифруются с использованием AES-256-GCM и сохраняются в `server/data/media/`.

**Важно**: Ключ шифрования должен быть ровно 32 байта (64 шестнадцатеричных символа) для шифрования AES-256.

Кроме того, вы можете настроить максимальный размер файла для загрузки, установив переменную окружения `CHAGOURTEE_MAX_FILE_SIZE` в байтах (по умолчанию 50MB = 52428800 байт).

Также можно настроить время жизни кэшированных аудио транскодов (в днях), установив переменную окружения `CHAGOURTEE_AUDIO_TRANSCODE_TTL_DAYS` (по умолчанию 0, что означает отсутствие срока истечения до удаления сообщения).
